# Template Schema Guide

A SuprSend template is a collection of **variants**. Each variant is a self-contained JSON document conforming to the variant schema, with its `content` validated against a per-channel schema chosen by the variant's `channel` field.

**Variant schema URL:** `https://schema.suprsend.com/template/v2/variant_schema.json`

## When is one variant enough?

If the template only needs to send to one channel, in one language, with no per-tenant or per-segment customization, you have **one variant**. Set `tenant_id: null`, pick a `locale` (typically `"en"`), and set `conditions: null`. The variant is still required â€” there is no schema for a "single content blob" outside the variant envelope.

You add more variants when:

- **Multiple channels** â€” one variant per channel (email + sms + inbox for the same template each gets its own variant).
- **Multiple locales** â€” one variant per `locale` per channel for translations.
- **Multi-tenant** â€” set `tenant_id` to a tenant slug to customize content per tenant/brand.
- **Conditional content** â€” use `conditions` to choose a variant based on recipient or payload attributes.

## Variant Envelope

| Property | Type | Required | Description |
| --- | --- | --- | --- |
| $schema | string | No |  |
| channel | string | Yes |  |
| id | string | Yes |  |
| tenant_id | string | null | Yes |  |
| locale | string | Yes | locale for variant |
| conditions | array | null | Yes | variant conditions |
| content | object | No |  |
| needs_vendor_approval | boolean | No | does content require vendor approval before sending. This is used for SMS content |
| seq_no | integer | null | No |  |
| vendor_approvals | array | No | approvals info with respect to variants that needs vendor approval |

### `channel` values

- `email`
- `sms`
- `whatsapp`
- `inbox`
- `slack`
- `ms_teams`
- `androidpush`
- `iospush`
- `webpush`

The `channel` selects which schema validates the `content` object. See the per-channel references for `content` structure.

### `id`

A unique identifier for the variant within the template â€” `^[a-zA-Z0-9_\-.]+$`, max 128 chars. Each variant must have a distinct `id`.

### `tenant_id`

`null` for non-tenant variants, or a tenant slug (`^[a-zA-Z0-9_\-]+$`, max 64 chars). When the template is rendered for a specific tenant, SuprSend prefers a matching `tenant_id` variant over the `null` (default) variant for the same channel/locale.

### `locale`

A 2â€“6 character locale code (e.g., `en`, `en-us`, `fr`, `hi-in`). At least one variant per channel should target the default locale you ship.

### `conditions`

`null` or an array of `expression_v1` clauses. When non-null, the variant is only used if all conditions evaluate to true for the current send. Useful for segmentation (e.g., a different subject line for enterprise users).

Each condition follows:

```json
{
  "type": "expression_v1",
  "expression_v1": {
    "op": "AND",
    "args": [
      { "variable_ns": "$recipient", "variable": "plan", "op": "==", "value": "enterprise" }
    ]
  }
}
```

#### Variable namespaces

| Namespace | Source |
|---|---|
| *(empty / omitted)* | Trigger event payload |
| `$actor` | Properties of the user who performed the action |
| `$recipient` | Properties of the user receiving the notification |
| `$brand` | Properties of the tenant/brand |
| `$organization` | Organization-level properties |

#### Comparison operators

- `==`
- `!=`
- `>`
- `>=`
- `<`
- `<=`
- `EMPTY`
- `NON_EMPTY`
- `CONTAINS`
- `NOT_CONTAINS`
- `ARRAY_INTERSECTS`
- `NOT_ARRAY_INTERSECTS`
- `DATETIME_EQUALS`
- `DATETIME_LT`
- `DATETIME_GT`

### `needs_vendor_approval` / `vendor_approvals`

Used for channels where the underlying vendor reviews content before send (notably SMS DLT and WhatsApp). When `needs_vendor_approval: true`, the template cannot be delivered until at least one `vendor_approvals` entry has `approval_status: "approved"`.

The `vendor_approvals` array entries follow:

| Property | Type | Required | Description |
| --- | --- | --- | --- |
| vendor_slug | string | Yes | vendor slug against which approval is done |
| vendor_uid | string | Yes | vendor unique-identifier against which approval is done |
| vendor_template_name | string | Yes | template-name set during vendor approval |
| vendor_template_id | string | null | No | template-id given by vendor after successful template upload |
| vendor_locale_code | string | null | No | language code addes during approval |
| vendor_template_category | string | null | No | in whatsapp, template category assigned after approval |
| provider_template_id | string | null | No | if msg91-sms vendor, set template_id generated by this vendor, in addition to DLT template-id |
| approval_status | string | Yes |  |
| comment | string | null | No | in case of rejection, this is rejection reason/comment |

For SMS variants, `needs_vendor_approval` is **required** (see the channel reference for details on DLT compliance).

### `seq_no`

Optional positive integer. When multiple variants match the same recipient (e.g., overlapping conditions), `seq_no` is used to order priority â€” lower numbers win.

## Variant Selection

When a workflow sends a template, SuprSend picks one variant per channel using:

1. **Channel match** â€” must equal the channel being sent.
2. **Tenant match** â€” prefer the variant whose `tenant_id` equals the send's tenant; otherwise fall back to `tenant_id: null`.
3. **Locale match** â€” prefer the recipient's locale; otherwise fall back to the template's default locale.
4. **Condition match** â€” only variants whose `conditions` evaluate to true are eligible.
5. **`seq_no` tiebreaker** â€” among remaining eligible variants, the lowest `seq_no` wins.

## When to add more variants

- **Per-tenant content** â€” see [Multi-tenant Variants](references/multi-tenant-variants.md) for how `tenant_id` lets you customize a template per tenant/brand.
- **Per-language content** â€” see [Multi-lingual Variants](references/multi-lingual-variants.md) for the trade-off between translation keys (one variant, many languages) vs. one variant per `locale`.

## Create a variant

Click the **+** button next to "All Variants" in the left panel. You'll see two options:


### Add a new channel

Select **New Channel** to add a channel to the template (for example, add SMS to a template that only had Email). Pick the channels and click **Add Channels**.


### Add a new variant

Select **New Variant** to create a variant with conditions for an existing channel.


Fill in:

**Channels** â€” which channels this variant applies to. *(Required)*

**ID** â€” a unique identifier. Use descriptive names: `uber-en`, `acme-es`, `pro-users`. *(Required)*

**Locale** â€” the language for this variant. *(Required)*

**Tenant** â€” scope to a specific tenant. Leave blank for non-tenant variants.

**Conditions** â€” rules that determine when this variant is sent. Each condition has a **Data Type** (Recipient, Actor, Tenant, or Input Payload), **Property**, **Operator**, and **Value**. Add multiple conditions with **+ OR**.

| Common operators                                         |                      |
| -------------------------------------------------------- | -------------------- |
| `==`, `!=`                                               | Equals / not equals  |
| `>`, `>=`, `<`, `<=`                                     | Comparisons          |
| `contains`, `not contains`                               | Substring match      |
| `is empty`, `is not empty`                               | Null checks          |
| `datetime is`, `datetime is before`, `datetime is after` | Date comparisons     |
| `intersects (array)`, `not intersects (array)`           | Array overlap checks |

* **Edit conditions** â€” click the **settings icon** on the variant to change its ID, locale, tenant, or conditions after creation.

## Commit

Click **Commit** to publish. The modal shows all changes â€” you can deselect specific variants you're not ready to publish.


For **WhatsApp** and **SMS (DLT)** variants, committing puts the variant into **Approval Pending** â€” it goes live only after vendor approval. For multi-lingual templates, each language is approved separately. If a [fallback vendor](/docs/vendor-fallback) is configured, you'll need to confirm approval for each vendor.

## How selection works at send time

When a notification is triggered, SuprSend evaluates all non-default variants in the order they appear in the panel. The first one whose conditions all match is sent. If no variant matches, the **default** (always at the top) is sent as the fallback.

* All conditions on a variant must be true â€” partial matches are skipped.
* **Tenant** and **locale** are soft checks with fallback: tenant+locale â†’ tenant+default locale â†’ default tenant+locale â†’ default.
* All other conditions (like `$recipient.plan == "pro"`) are hard checks â€” no fallback, the variant is just skipped.


### Ordering variants

Order determines priority. If two variants both match a recipient's data, the one **higher in the list** wins. You can drag and drop variants to change their position.

* The **default** variant always stays at the top â€” it's the fallback and cannot be moved. No other variant can be placed above it.
* All other variants can be freely reordered by dragging.
* Place more specific variants (for example, `uber-es`) above general ones (for example, `default-es`) to ensure they match first.

If a variant isn't being picked when you expect it to, check its position â€” a broader variant above it may be matching first.

### Testing variant selection

To verify the right variant is being picked for a specific condition, **test the full template group** â€” not a single variant. Testing a single variant sends that variant's content directly (bypassing conditions). Testing the template group runs the actual selection logic:

1. Click **Test** â†’ select the **Template Group** tab.
2. Enter a **Recipient** whose profile matches the condition you want to test.
3. Select a **Tenant** if your variant has a tenant condition.
4. Click **Send Test** â€” SuprSend evaluates all variant conditions and sends the matching one.
5. Check [Logs](/docs/logging) to confirm which variant was selected.

This is the only way to verify that ordering and conditions work as expected end-to-end.

## Duplicate, delete, and disable

### Duplicate a variant

Click the three-dot menu (**...**) on any variant to duplicate it. The content is copied into a new variant â€” adjust the ID and conditions.


### Delete a variant

Click the three-dot menu on a variant and select **Delete**. This permanently removes the variant and its content. The **default** variant cannot be deleted.

### Disable a channel

Click the three-dot menu on a **channel name** (not a variant) to disable it. Notifications stop on that channel immediately, but the content is preserved. Re-enable anytime without losing work.


This is useful when you want to temporarily pause a channel (for example, stop sending SMS during a vendor migration) without deleting the content and having to recreate it later.

## Common patterns

**One template, many tenants** â€” create a variant per tenant using a `$tenant.id` condition. Default is the fallback for tenants without a custom variant. Onboarding a new tenant = a template edit, no code change.

**Tenant + language** â€” combine conditions: `$tenant.id = acme` + `$recipient.language = fr`. Place these above language-only variants so they match first.

**A/B testing** â€” create a variant targeting a subset (for example, `$recipient.cohort = beta`). Compare metrics. To promote: widen the conditions or update the default. To end: delete the challenger.

After setting up any pattern, [test with a specific tenant and recipient](/docs/templates#test) to verify the right variant is picked.

## Frequently asked questions


  ### What happens if no variant matches?
    The **default** variant is sent. It cannot be deleted and always acts as the fallback.
  

  ### Can I have different variants per channel?
    Yes. Variants are per-channel â€” you can have 3 for Email, 1 for SMS, and just the default for Push. Each channel is evaluated independently.
  

  ### How do I test which variant gets picked?
    Click **Test** in the editor, enter a recipient, select a tenant, and send. The variant whose conditions match is used. Check [Logs](/docs/logging) to confirm.
  

  ### What's the difference between soft and hard conditions?
    **Soft** (tenant, locale): SuprSend falls back through a hierarchy if no exact match exists. **Hard** (everything else): no fallback â€” the variant is skipped if the condition doesn't match.
  

  ### Can I reorder variants?
    Yes. Drag them up or down in the panel. The default always stays at the top and cannot be moved.
